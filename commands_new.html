Module LINEAR CURVE

Description:

This module includes an efficient single-dimentional indexing module with the same API like Morton curve. This structure allows you to store and query points in 1D by mapping.

It is made for completeness.

Coordinate are uint32_t.

See Also:

MC1POINT

Command MC1POINT
Available since 1.3.7.7
Data lookup complexity READ
Redis compatible N
Mutable N
Module LINEAR CURVE

Usage:

MC1POINT key x y count start

Parameters:

key name of the Linear curve object
x coordinate of the stored point / element of the stored vector. Coordinate is uint32_t.
count result count
start specify starting key (for pagination in the similar way as in XNGET)

Description:

Get all elements from Linear curve index *key*, that are in the "point" with "coordinates" *x*.
... or ...
In other words, get all vectors from Linear curve index *key*, that have elements { *x* }.

This command is way faster than MC1RANGE.

Example usage:

Suppose we have a company with two offices in:

  • LA = 1
  • NY = 2
mc1add users 0 1 john   <- adds user_id 0, john,   LA
mc1add users 1 1 peter  <- adds user_id 1, peter,  LA
mc1add users 2 2 jill   <- adds user_id 2, jill,   LA
mc1add users 3 1 robert <- adds user_id 3, robert, NY
mc1add users 4 2 pola   <- adds user_id 4, pola,   NY
mc1add users 5 2 pepe   <- adds user_id 5, pepe,   NY
mc1point users 2 2 10000  <- gets all support from LA (pola, pola, pepe)

Return type:

array

Return value:

list of key / values with "next" key at the end.

Example:

127.0.0.1:6379> MC1ADD users 0 1 john
OK
127.0.0.1:6379> MC1ADD users 1 1 peter
OK

Module MORTON CURVE 2D

Description:

This module includes an efficient multidimensional indexing module based on the Morton curve (also known as Z-order curve). This structure allows you to store and query points in 2D by mapping them into a single sortable integer using bit interleaving.

Coordinates are uint32_t, indexes type are uint64_t

The result is a fast, disk-friendly spatial index suitable for filtering and range searches.

General Concept

  • You insert a point with some coordinates.
  • The module computes the Morton key by interleaving bits.
  • All points are stored in sorted order by their Morton code.
  • Queries operate by converting coordinates or ranges into corresponding Morton code intervals

All MC commands share the same pattern: MC{n}{op}
Where:

  • n = number of dimensions (2, 3, 4, 8, 16)
  • op = operation: ADD, POINT, RANGE, etc.

Notes

  • Morton curves preserve spatial locality well, making disk-based range lookups efficient.
  • POINT = optimized mode when all dimensions are exact values.
  • RANGE = full multidimensional "rectangle".
  • Coordinates are uint32_t
  • Lndexes type are uint64_t

Example usage:

Suppose we have a company with two offices in:

  • LA = 1
  • NY = 2

Each office have two kind of employees:

  • sales = 1
  • support = 2
mc2add users 0 1 1 john      <- adds user_id 0, john,   sales,   LA
mc2add users 1 1 1 peter     <- adds user_id 1, peter,  sales,   LA
mc2add users 2 2 1 jill      <- adds user_id 2, jill,   support, LA
mc2add users 3 1 2 robert    <- adds user_id 3, robert, sales,   NY
mc2add users 4 2 2 pola      <- adds user_id 4, pola,   support, NY
mc2add users 5 2 2 pepe      <- adds user_id 5, pepe,   support, NY
mc2point users 2 2 10000     <- gets all support from LA               (pola, pepe)
mc2point users 2 2 10000     <- gets all support from LA               (pola, pepe)
mc2range users 2 2 2 2 10000 <- gets all support from LA               (pola, pepe) - same as previous, but slower.
mc2range users 2 2 1 2 10000 <- gets support from LA and NY            (jill, pola, pepe)
mc2range users 1 2 2 2 10000 <- gets sales and support from LA         (robert, pola, pepe)
mc2range users 1 2 1 2 10000 <- gets sales and support from LA and NY  (all)

Module MORTON CURVE 3D

Description:

This module includes an efficient multidimensional indexing module based on the Morton curve (also known as Z-order curve). This structure allows you to store and query points in 3D by mapping them into a single sortable integer using bit interleaving.

Coordinates are uint32_t, indexes type are uint128_t

The result is a fast, disk-friendly spatial index suitable for filtering and range searches.

General Concept

  • You insert a point with some coordinates.
  • The module computes the Morton key by interleaving bits.
  • All points are stored in sorted order by their Morton code.
  • Queries operate by converting coordinates or ranges into corresponding Morton code intervals

All MC commands share the same pattern: MC{n}{op}
Where:

  • n = number of dimensions (2, 3, 4, 8, 16)
  • op = operation: ADD, POINT, RANGE, etc.

Notes

  • Morton curves preserve spatial locality well, making disk-based range lookups efficient.
  • POINT = optimized mode when all dimensions are exact values.
  • RANGE = full multidimensional "rectangle".
  • Coordinates are uint32_t
  • Lndexes type are uint128_t

Module MORTON CURVE 8D

Description:

This module includes an efficient multidimensional indexing module based on the Morton curve (also known as Z-order curve). This structure allows you to store and query points in 8D by mapping them into a single sortable integer using bit interleaving.

Coordinates are uint8_t, indexes type are uint64_t

The result is a fast, disk-friendly spatial index suitable for filtering and range searches.

General Concept

  • You insert a point with some coordinates.
  • The module computes the Morton key by interleaving bits.
  • All points are stored in sorted order by their Morton code.
  • Queries operate by converting coordinates or ranges into corresponding Morton code intervals

All MC commands share the same pattern: MC{n}{op}
Where:

  • n = number of dimensions (2, 3, 4, 8, 16)
  • op = operation: ADD, POINT, RANGE, etc.

Notes

  • Morton curves preserve spatial locality well, making disk-based range lookups efficient.
  • POINT = optimized mode when all dimensions are exact values.
  • RANGE = full multidimensional "rectangle".
  • Coordinates are uint8_t
  • Lndexes type are uint64_t

  • Queries operate by converting coordinates or ranges into corresponding Morton code intervals

All MC commands share the same pattern: MC{n}{op}
Where:

  • n = number of dimensions (2, 3, 4, 8, 16)
  • op = operation: ADD, POINT, RANGE, etc.

Notes

  • Morton curves preserve spatial locality well, making disk-based range lookups efficient.
  • POINT = optimized mode when all dimensions are exact values.
  • RANGE = full multidimensional "rectangle".
  • Coordinates are uint8_t
  • Lndexes type are uint128_t

Module HEAVY HEATERS

Description:

This module implements simple heavy hitters algorithm, allowing you to track top items in a fixed-size collection efficiently.

How It Works:

1. Adding items

Use the HHINCR (if higer score is better) or HHDECR (if lower score is better) command to add one or more values into the heavy hitters object. The object has a fixed size (hh_count) and each value has a fixed byte size (hh_value_size).

2. Supports fixed-size values

Each item stored in the heavy hitters has a predefined size (hh_value_size).

This keeps the structure compact and guarantees stable memory usage.

Module RESERVOIR SAMPLING

Description:

This module implements reservoir sampling — a technique used to keep a uniformly random sample from a stream of values, even when the total number of items is unknown or extremely large.

What It Does:

1. Stores a fixed-size random sample

You can add any number of values to a reservoir sampling object using RSADD. Even if millions of items are added, the module keeps only rs_count items, but ensures that every item from the stream has an equal chance to appear in the reservoir.

2. Supports fixed-size values

Each item stored in the reservoir has a predefined size (rs_value_size). This keeps the structure compact and guarantees stable memory usage.

512 510Accepted after 1.3.15.1

3. Randomized insertion logic

When adding items, some values will replace existing samples while others will be skipped. RSADD returns:

  • 1 - if at least one of the provided items was included in the reservoir
  • 0 - if none of the items were selected

4. Retrieving the sample

You can retrieve the current sample set using RSGET. The returned array contains up to rs_count items.

5. Tracking total insertions

RSGETCOUNT tells you how many total items were attempted for insertion into the reservoir, even if they were not selected.

Use Cases:

  • Analytics pipelines
  • Approximation algorithms
  • Monitoring and logging systems

Module RING_BUFFER

Description:

This module implements a high-performance ring buffer (circular queue) - a data structure that uses a single, fixed-size block of memory as if it were connected end-to-end. It is highly optimized for fast FIFO (First-In, First-Out) data streaming.

What It Does:

1. Fixed-size circular storage with auto-overwrite

You can push any number of values into the ring buffer using RBADD (alias RBPUSH). Once the buffer reaches its maximum capacity (rb_count), new incoming items automatically overwrite the oldest ones. This completely eliminates the need for shifting elements or allocating memory at runtime.

2. Supports fixed-size values

Each item stored in the buffer maps to a predefined slot size (rb_value_size). This guarantees zero memory fragmentation and deterministic performance under heavy loads.

64 64
128 127
256 255Pascal string
512 510
10241022

3. Multi-item insertion

RBADD supports batch insertion, allowing multiple values to be pushed in a single command. It returns 1 upon successful processing.

4. Non-destructive peeking

You can retrieve the current active items in their exact chronological order using RBGET without removing them.

5. Destructive popping (FIFO)

RBPOP extracts and returns the oldest active item from the buffer, advancing the internal head pointer. If the buffer is empty, it safely returns an empty string ("") to prevent underflow errors.

6. Tracking active elements

RBCOUNT (alias RBLEN) returns the exact number of active elements currently held in the buffer (ranging from 0 up to rb_count).

Use Cases:

  • Sliding window logging and telemetry tracking
  • High-throughput message queues and streaming buffers
  • Time-series metrics caching
  • Decoupling fast producers from slower asynchronous consumers

Module STRING_LIST

Description:

This module implements a high-performance packed string list - a data structure that sequentially stores variable-length strings into a contiguous block of memory. It is highly optimized for extreme memory efficiency and fast index-based retrieval.

What It Does:

1. Sequential appending with smart capacity

You can push any number of variable-length strings into the list using SLADD (aliase SLPUSH). This command may reserve a slight memory buffer (advance capacity) under the hood to make subsequent insertions incredibly fast without constantly reallocating memory.

2. Exact-fit memory compaction

For strictly memory-constrained scenarios or finalized lists, SLADDPACK (aliases SLADDSHORT, SLPUSHPACK, SLPUSHSHORT) adds the new elements and tightly compacts the memory footprint exactly to the required bytes, leaving absolutely zero wasted space.

3. Multi-item insertion

Both insertion commands support batch appending, allowing multiple strings to be pushed in a single command. They return 1 upon successful processing.

4. Fast single and multiple index retrieval

You can retrieve specific items by their 0-based chronological index using SLGET. For bulk fetching, SLMGET accepts multiple indices and uses a highly optimized internal query planner to resolve the request instantly. Out-of-bounds indices safely return an empty string ("").

5. Complete list retrieval

SLGETALL extracts and returns all elements currently stored in the list in their exact chronological order of insertion.

6. Tracking list size

SLCOUNT returns the exact number of elements currently stored in the list.

Use Cases:

  • Tracking user behavior or history trails (e.g., sequentially visited URLs or IPs)
  • Memory-restricted data aggregation
  • Storing tightly packed, sequential, variable-length events

Tip: To clear or reset the list completely, use the global DEL command on the key. This will instantly release the allocated memory.

Example:

127.0.0.1:6379> sladd a Germany USA
(integer) 1
127.0.0.1:6379> sladdpack a UK Denmark
(integer) 1
127.0.0.1:6379> slcount a
(integer) 4
127.0.0.1:6379> slget a 3
"Denmark"
127.0.0.1:6379> slmget a 0 3 102
1) "Germany"
2) "Denmark"
3) ""
127.0.0.1:6379> slgetall a
1) "Germany"
2) "USA"
3) "UK"
4) "Denmark"

Module VECTORS

Description:

HM4 can be used as a vector database. Unlike most vector DBs, HM4 stores data on disk, so memory-heavy algorithms like HNSW are impractical. Instead HM4 provides two alternatives: Flat Search (exact) and Locality-Sensitive Hashing (LSH) (approximate).

Modes of Operation:

  • With FLAT/LSH indexing: use VADD, VSIMFLAT, VSIMLSH.
  • With user key-only storage: use VKSET, VKSIMFLAT. In this mode LSH indexing is not available.

Building a Vector Index (VADD)

Use VADD to create a vector index.

Parameters: input dimensionality, stored dimensionality (projection), quantization type, raw vector format, then pairs of BLOB name.

Searching a Vector Index

Use VSIMFLAT for exact (brute-force) search and VSIMLSH for fast approximate search.

Distance / Similarity Metrics :

Flag Metric Name Range Bit Description When to use
E Euclidean (L2) 0 .. ∞   Standard straight-line distance between two points in Euclidean space. Best for dense, normalized data where overall magnitude matters. It’s the default in many ML applications.
M Manhattan (L1) 0 .. ∞   Sum of absolute differences across dimensions. Also known as "taxicab" or "city-block" distance. More robust to outliers; works well when vector components are sparse or vary in only a few dimensions.
C Cosine 0 .. 1   Measures the cosine of the angle between two vectors (orientation, not magnitude). Focuses on direction, not magnitude. Ideal for text embeddings or any case where you care about angular similarity rather than size.

H Hamming 0 .. 1 Yes Works only on bit quantized vectors. Works only on bit quantized vectors. Similar to hamming, but result is "weighted" to be more "human understandable".

Additional Vector Commands

  • VREM key name - remove vector with given key from index.
  • VGET key dim type name - get vector in human-readable form.
  • VGETNORMALIZED - return normalized vector with magnitude.
  • VGETRAW - get raw binary/hex representation.

Key-Based Vectors (no LSH)

Store vectors directly under keys (no index) with VKSET and query with VKSIMFLAT to search over keys sharing a prefix.

Notes & Tips

  • Flat searches are exact but slower; use for small indexes or when accuracy is critical.
  • LSH trades accuracy for speed.
  • Choose distance metric (E, M, C, K, H, B, D) according to data type and semantics.

Module MULTI INDEX

Description:

This module provides a multi-index object that helps in storing and retrieving data efficiently for your application, using word index.

How It Works:

1. Storing entries with multiple sub-keys

You can insert entries into a mindex object using IXMADD. Each entry can have a sub_key, sort_key, and multiple indexes (words). Example:

  • IXMADD mykey John , "london,new york" x "Data for John" - stores data under sub_key "John" with indexes "london" and "new york".

2. Retrieving values

You can get values from the mindex object using IXMGET or IXMMGET. Single or multiple sub-keys can be queried. Example:

  • IXMGET mykey John - retrieves the value for sub_key "John".
  • IXMMGET mykey John Pete Ivan - retrieves values for several sub_keys.

3. Checking existence

Use IXMEXISTS to check whether a value exists for a given sub_key. Example:

  • IXMEXISTS mykey John - returns 1 if exists, 0 otherwise.

4. Removing entries

Remove entries using IXMREM (aliases: IXMREMOVE, IXMDEL). Example:

  • IXMREM mykey John - deletes the entry for sub_key "John".

5. Fetching ranges

You can get a range of indexed results using IXMRANGE, IXMRANGEFLEX, and IXMRANGESTRICT. These commands allow pagination and filtering based on indexes. Example:

  • IXMRANGE mykey london 100 '' - fetches up to 100 results starting from key "london".
  • IXMRANGEFLEX mykey , 'london,new york' 100 '' - returns results that match any of the given indexes.
  • IXMRANGESTRICT mykey , 'london,new york' 100 '' - returns results only if all indexes match.

Tip: This module can be used both for tagging and full-text search. Always add your data correctly before querying ranges to ensure accurate results.

Module TRIGRAM INDEX

Description:

This module provides a multi-index object that helps in storing and retrieving data efficiently for your application, using trigram index.

How It Works:

1. Storing entries with multiple sub-keys

You can insert entries into a mindex object using IXMADD. Each entry can have a sub_key, sort_key, and multiple indexes (words). Example:

  • IXMADD mykey John , "london,new york,köln,hi" x "Data for John" - stores data under sub_key "John" with indexes "london" and "new york".

Instead of storing "london" and "new york", this index stores trigrams for each word, e.g.

"london" "lon", "ond", "ndo", "don"  
"new york" "new", "ew_", "w_y", "_yo", "yor", "ork" The spaces are replaced with underscores for the representation. Real trigrams include spaces.
"köln" "köl", "öln" IXMADD is UTF8 aware, it tokenize the trigrams correctly.
IXMADD_BIN can be used for old style 8 bit encodings like CP1250, CP1251, KOI8 etc.
"hi" "hi" If the word is not 3 symbols, it is stored in the index as is.

2. Retrieving values

You can get values from the mindex object using IXMGET or IXMMGET. Single or multiple sub-keys can be queried. Example:

  • IXMGET mykey John - retrieves the value for sub_key "John".
  • IXMMGET mykey John Pete Ivan - retrieves values for several sub_keys.

3. Checking existence

Use IXMEXISTS to check whether a value exists for a given sub_key. Example:

  • IXMEXISTS mykey John - returns 1 if exists, 0 otherwise.

4. Removing entries

Remove entries using IXMREM (aliases: IXMREMOVE, IXMDEL). Example:

  • IXMREM mykey John - deletes the entry for sub_key "John".

5. Fetching ranges

You can get a range of indexed results using IXMRANGE, IXMRANGEFLEX, and IXMRANGESTRICT. These commands allow pagination and filtering based on indexes. Example:

  • IXMRANGEFLEX mykey , 'london,new york' 100 '' - returns results that match any of the given indexes.
  • IXMRANGESTRICT mykey , 'london,new york' 100 '' - returns results only if all indexes match.

Tip: This module can be used both for tagging and full-text search. Always add your data correctly before querying ranges to ensure accurate results.

Module ISAM

Description:

This module provides commands for working with structured objects stored as a single, space-padded string of fixed-width fields. It mimics a lightweight key-value or hash structure without the memory overhead of storing field names inside every record.

How It Works:

1. Dynamic Schema Definition

Every command requires a schema definition passed as its first argument. The schema format is: size:field_name,size:field_name,.... Sizes are evaluated in bytes. If a value is shorter than the allocated size, it is padded with spaces. If it is longer, it gets truncated automatically.

2. Modifying fields

The module allows setting all fields at once or updating specific fields selectively without altering the rest of the string layout.

  • isetall - overwrites or initializes all fields sequentially based on the schema layout.
  • iset - selectively updates only the specified fields at their exact byte offsets.
  • idel - "clears" the specified fields by replacing their contents with space padding.

3. Retrieving data

Data can be fetched as flat lists of key-value pairs, standalone keys, or standalone values. All returned values are automatically stripped of their trailing space padding.

  • igetall - returns an array of all field names and their corresponding values (similar to HGETALL).
  • igetkeys - returns an array containing only the field names.
  • igetvals - returns an array containing only the actual data values.
  • imget - fetches values for specific requested fields in any order, supporting duplicates and returning an empty string for non-existent fields.

4. UTF-8 and Multi-byte Characters

Since field sizes are strictly byte-based, multi-byte characters (like Cyrillic UTF-8) consume 2 to 4 bytes per character. It is recommended to place variable-length UTF-8 text fields at the end of the schema to prevent unintended byte truncation or offsetting.

Tip: Use igetall when you need a full object dump, and imget to quickly cherry-pick specific fields in a single network round-trip.

Module summary_stats

Description:

This module provides commands for working with online stream statistics over numeric data sets. It allows high-throughput, memory-efficient accumulation of data points and extracts multi-variable statistical properties without storing individual samples.

How It Works:

1. Fixed-Size Statistical Accumulator

Every statistics key allocates a fixed-size internal structure (AccData) that tracks counts, chronological markers, extremes, and sum products. Statistical operations run strictly inside a pre-allocated memory pool with zero heap allocations during query execution.

Data can be pushed into an accumulator individually or combined from multiple separate historical streams.

For high-frequency logging streams, the module offers combined commands that accept new data points and immediately return updated results in a single, atomic network transaction.

  • ssaddgetall - pushes values and instantly returns the full updated 38-element statistical report.
  • ssaddget - pushes a value and instantly returns only the requested single metric.
  • ssaddmget - pushes a value and instantly returns an array of specific requested subkeys.

Module MURMUR

Description:

This module provides fast and reliable non-cryptographic hashing using the well-known MurmurHash64A algorithm.

What It Does:

1. Generates a 64-bit hash from any string

The command MURMUR takes an input string and produces a deterministic 64-bit hash value. This type of hash is ideal for systems where speed and distribution quality are more important than cryptographic security.

2. Optional seed value

You can provide a custom seed to influence the hash outcome. Different seeds generate different hash outputs for the same string, which is useful for sharding, randomized hashing, and avoiding collisions in custom scenarios.

3. Optional modulo reduction

If you include a modulo argument, the result will be reduced to hash % modulo and returned as an integer. This is particularly useful for:

  • distributing keys across buckets,
  • implementing partitioned storage,
  • mapping hashes into a fixed numeric range.

Use Cases:

  • Fast indexing
  • Hash-based partitioning
  • Deduplication checks
  • Bloom filters and probabilistic data structures

Note: MurmurHash is not cryptographically secure and should not be used for password hashing or security-critical operations.

Example:

127.0.0.1:6379> murmur test

127.0.0.1:6379> murmur test 0

127.0.0.1:6379> murmur test 1

Module RANDOM

Description:

This module provides commands for generating pseudo-random numbers. The numbers are deterministic: the same seed always produces the same result.

How It Works:

1. Generating a random number

The RANDOM command returns a single pseudo-random number using murmur hash mixer. If no seed is given, an internal seed (equal to zero) is used. If you provide a seed, the output is always the same for that seed.

  • RANDOM - returns the number generated from seed 0. .
  • RANDOM 5 - returns the number generated from seed 5.

2. Generating multiple random numbers

The MRANDOM command accepts multiple seeds and returns an array of random numbers, one for each seed.

  • MRANDOM 0 1 2 3 - returns four pseudo-random numbers, each derived from the respective seed.

3. Deterministic output

The generator is fully deterministic: the same seed always maps to the same output. This makes the module useful for testing, simulations, sharding, or any logic where reproducible randomness is required.

4. Output types

Depending on the internal state, RANDOM may return a string or an integer. MRANDOM always returns an array of such values.

Tip: Use MRANDOM when you need several reproducible values at once. Use RANDOM without arguments when you simply need a single pseudo-random value.
