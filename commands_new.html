Module MORTON CURVE 2D

Description:

This module includes an efficient multidimensional indexing module based on the Morton curve (also known as Z-order curve). This structure allows you to store and query points in 2D by mapping them into a single sortable integer using bit interleaving.

Coordinates are uint32_t, indexes type are uint64_t

The result is a fast, disk-friendly spatial index suitable for filtering and range searches.

General Concept

  • You insert a point with some coordinates.
  • The module computes the Morton key by interleaving bits.
  • All points are stored in sorted order by their Morton code.
  • Queries operate by converting coordinates or ranges into corresponding Morton code intervals

All MC commands share the same pattern: MC{n}{op}
Where:

  • n = number of dimensions (2, 3, 4, 8, 16)
  • op = operation: ADD, POINT, RANGE, etc.

Notes

  • Morton curves preserve spatial locality well, making disk-based range lookups efficient.
  • POINT = optimized mode when all dimensions are exact values.
  • RANGE = full multidimensional "rectangle".
  • Coordinates are uint32_t
  • Lndexes type are uint64_t

Suppose we have a company with two offices in:

  • LA = 1
  • NYC = 2

Each office have two kind of employees:

  • sales = 1
  • support = 2
mc2add users 0 1 1 john   <- adds user_id 0, john,   sales,   LA
mc2add users 1 1 1 peter  <- adds user_id 1, peter,  sales,   LA
mc2add users 2 2 1 jill   <- adds user_id 2, jill,   support, LA
mc2add users 3 1 2 robert <- adds user_id 3, robert, sales,   NYC
mc2add users 4 2 2 pola   <- adds user_id 4, pola,   support, NYC
mc2add users 5 2 2 pepe   <- adds user_id 5, pepe,   support, NYC
mc2point users 2 2 10000  <- gets all support from LA (pola, pepe)

Module HEAVY HEATERS

Description:

This module implements simple heavy hitters algorithm, allowing you to track top items in a fixed-size collection efficiently.

How It Works:

1. Adding items

Use the HHINCR (if higer score is better) or HHDECR (if lower score is better) command to add one or more values into the heavy hitters object. The object has a fixed size (hh_count) and each value has a fixed byte size (hh_value_size).

2. Supports fixed-size values

Each item stored in the heavy hitters has a predefined size (hh_value_size).

This keeps the structure compact and guarantees stable memory usage.

ValueSizeComment
16 15IP4
32 31 
40 39IP6
64 64
128127
256255Pascal string

3. Getting top items

Use the HHGET command to retrieve the top items and their counts from the object.

Tip: This module is ideal for tracking frequent items in a stream, such as popular searches, tags, or event identifiers. However unlike similar modules such Misra-Gries heavy hitters, you have to supply the score. See the examples for some ideas.

Example:

(integer) 1
127.0.0.1:6379> hhincr myhh 5 64 google 140 yahoo 34 cloudflare 13 outlook 10 facebook 5
(integer) 1

Module RESERVOIR SAMPLING

Description:

This module implements reservoir sampling â€” a technique used to keep a uniformly random sample from a stream of values, even when the total number of items is unknown or extremely large.

What It Does:

1. Stores a fixed-size random sample

You can add any number of values to a reservoir sampling object using RSADD. Even if millions of items are added, the module keeps only rs_count items, but ensures that every item from the stream has an equal chance to appear in the reservoir.

2. Supports fixed-size values

Each item stored in the reservoir has a predefined size (rs_value_size). This keeps the structure compact and guarantees stable memory usage.

ValueSizeComment
16 15IP4
32 31 
40 39IP6
64 64
128127
256255Pascal string

3. Randomized insertion logic

When adding items, some values will replace existing samples while others will be skipped. RSADD returns:

  • 1 - if at least one of the provided items was included in the reservoir
  • 0 - if none of the items were selected

4. Retrieving the sample

You can retrieve the current sample set using RSGET. The returned array contains up to rs_count items.

5. Tracking total insertions

RSGETCOUNT tells you how many total items were attempted for insertion into the reservoir, even if they were not selected.

Use Cases:

  • Uniform random sampling of high-volume data streams
  • Analytics pipelines
  • Approximation algorithms
  • Monitoring and logging systems

Tip: The reservoir sampling object is deterministic given identical sequences of inserts, making it useful for reproducible sampling scenarios.

Module VECTOR

Description:

HM4 can be used as a vector database. Unlike most vector DBs, HM4 stores data on disk, so memory-heavy algorithms like HNSW are impractical. Instead HM4 provides two alternatives: Flat Search (exact) and Locality-Sensitive Hashing (LSH) (approximate).

Modes of Operation:

  • With FLAT/LSH indexing: use VADD, VSIMFLAT, VSIMLSH.
  • With user key-only storage: use VKSET, VKSIMFLAT. In this mode LSH indexing is not available.

Building a Vector Index (VADD)

Use VADD to create a vector index.

Searching a Vector Index

Use VSIMFLAT for exact (brute-force) search and VSIMLSH for fast approximate search.

  • VGET key dim type name - get vector in human-readable form.

Module MINDEX

Description:

This module provides a multi-index object that helps in storing and retrieving data efficiently for your application.

How It Works:

1. Storing entries with multiple sub-keys

You can insert entries into a mindex object using IXMADD. Each entry can have a sub_key, sort_key, and multiple indexes (words). Example:

  • IXMADD mykey John , "london,new york" x "Data for John" - stores data under sub_key "John" with indexes "london" and "new york".

2. Retrieving values

You can get values from the mindex object using IXMGET or IXMMGET. Single or multiple sub-keys can be queried. Example:

  • IXMGET mykey John - retrieves the value for sub_key "John".
  • IXMMGET mykey John Pete Ivan - retrieves values for several sub_keys.

3. Checking existence

Use IXMEXISTS to check whether a value exists for a given sub_key. Example:

  • IXMEXISTS mykey John - returns 1 if exists, 0 otherwise.

4. Removing entries

Remove entries using IXMREM (aliases: IXMREMOVE, IXMDEL). Example:

  • IXMREM mykey John - deletes the entry for sub_key "John".

5. Fetching ranges

You can get a range of indexed results using IXMRANGE, IXMRANGEFLEX, and IXMRANGESTRICT. These commands allow pagination and filtering based on indexes. Example:

  • IXMRANGE mykey london 100 '' - fetches up to 100 results starting from key "london".
  • IXMRANGEFLEX mykey , 'london,new york' 100 '' - returns results that match any of the given indexes.
  • IXMRANGESTRICT mykey , 'london,new york' 100 '' - returns results only if all indexes match.

Tip: This module can be used both for tagging and full-text search. Always add your data correctly before querying ranges to ensure accurate results.

Example:

Module AUTOCOMPLETE

Description:

This module collects search strings and helps in building autocomplete functionality for your website or application.

How It Works:

1. Collecting valid search strings

The module stores "correct" search strings that users perform. For example, on a music-related website:

  • User searches for "madonna" - a result is found - this is a valid search string.
  • User searches for "madona" - no result is found - this is an invalid search string.

2. Adding valid search strings

Before the module can generate autocomplete suggestions, you need to add valid search strings using the command ACADD. Example: ACADD mykey "madonna" 3600. This tells the system: "this is a valid search string, store it for autocomplete."

3. Retrieving autocomplete results

Once the valid strings are added, you can use the command ACRANGE to get autocomplete suggestions. Example: ACRANGE mykey "ma". The system returns a list of suggestions starting with "ma".

4. Ordering results

Results are sorted by number of hits. More frequently used or more correct search strings appear first.

Tip: Always add valid search strings before using autocomplete. The commands ACADD and ACRANGE work for any type of search, not only music-related queries.

Example:

127.0.0.1:6379> ACADD mykey "hello sir"   3600
(integer) 1
127.0.0.1:6379> ACADD mykey "hello world" 3600

Module MURMUR

Description:

This module provides fast and reliable non-cryptographic hashing using the well-known MurmurHash64A algorithm.

What It Does:

1. Generates a 64-bit hash from any string

The command MURMUR takes an input string and produces a deterministic 64-bit hash value. This type of hash is ideal for systems where speed and distribution quality are more important than cryptographic security.

2. Optional seed value

You can provide a custom seed to influence the hash outcome. Different seeds generate different hash outputs for the same string, which is useful for sharding, randomized hashing, and avoiding collisions in custom scenarios.

3. Optional modulo reduction

If you include a modulo argument, the result will be reduced to hash % modulo and returned as an integer. This is particularly useful for:

  • distributing keys across buckets,
  • implementing partitioned storage,
  • mapping hashes into a fixed numeric range.

Use Cases:

  • Fast indexing
  • Hash-based partitioning
  • Deduplication checks
  • Bloom filters and probabilistic data structures

Note: MurmurHash is not cryptographically secure and should not be used for password hashing or security-critical operations.

Example:

127.0.0.1:6379> murmur test
"3407684658384555107"

Module RANDOM

Description:

This module provides commands for generating pseudo-random numbers. The numbers are deterministic: the same seed always produces the same result.

How It Works:

1. Generating a random number

The RANDOM command returns a single pseudo-random number using murmur hash mixer. If no seed is given, an internal seed (equal to zero) is used. If you provide a seed, the output is always the same for that seed.

  • RANDOM - returns the number generated from seed 0. .
  • RANDOM 5 - returns the number generated from seed 5.

2. Generating multiple random numbers

The MRANDOM command accepts multiple seeds and returns an array of random numbers, one for each seed.

  • MRANDOM 0 1 2 3 - returns four pseudo-random numbers, each derived from the respective seed.

3. Deterministic output

The generator is fully deterministic: the same seed always maps to the same output. This makes the module useful for testing, simulations, sharding, or any logic where reproducible randomness is required.

4. Output types

Depending on the internal state, RANDOM may return a string or an integer. MRANDOM always returns an array of such values.

Tip: Use MRANDOM when you need several reproducible values at once. Use RANDOM without arguments when you simply need a single pseudo-random value.

Example:

127.0.0.1:6379> random
"15926910803142346387"
127.0.0.1:6379> random 0
"15926910803142346387"
127.0.0.1:6379> random 1
